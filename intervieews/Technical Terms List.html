🧩 Product & Feature Changes

  • 🆕 New Feature — Introduces a net-new user-visible capability.

  • ⭐ Feature Enhancement — Extends or improves an existing feature.

  • 🧪 Experimental Feature — Feature behind flags, not production-stable.

  • 🧬 Feature Toggle — Adds or modifies runtime feature flags.

  • 🧱 Backward-Compatible Change — Adds capability without breaking APIs.

  • 💥 Breaking Change — Introduces incompatible API or behavior changes.

  • 🗑 Feature Deprecation — Marks functionality for future removal.

  • 🛑 Feature Removal — Deletes deprecated or obsolete functionality.

🐞 Bug & Reliability

  • 🐞 Bug Fix — Corrects incorrect or unintended behavior.

  • 🔥 Critical Fix — Fixes production-blocking or data-loss issues.

  • 🧯 Hotfix — Emergency fix applied directly to production.

  • 🧨 Race Condition Fix — Resolves concurrency timing issues.

  • 🧠 Logic Correction — Fixes flawed business or control logic.

  • 🔄 Regression Fix — Fixes issues reintroduced by previous changes.

  • 🧵 Thread Safety Fix — Eliminates unsafe shared-state access.

⚡ Performance & Efficiency

  • ⚡ Performance Optimization — Reduces latency or execution time.

  • 🗜 Memory Optimization — Reduces heap usage or memory leaks.

  • 🚀 Startup Time Reduction — Improves application boot speed.

  • 🔍 Query Optimization — Improves database or search queries.

  • 🛠 Refactoring — Improves internal structure without behavior change.

🔥 Incident Postmortem Checklist

1) Incident Metadata

  • ☐ Incident ID

  • ☐ Date / duration

  • ☐ Severity level

  • ☐ Affected systems

  • ☐ Customer impact summary

  • ☐ Incident owner

2) Detection & Alerting

  • ☐ How was it detected (alert, user, internal)?

  • ☐ Detection time vs actual start time

  • ☐ Alert quality (signal vs noise)

  • ☐ Missing or late alerts identified

  • ☐ Correct SLO/SLA breached?

3) Impact Assessment

  • ☐ User-facing symptoms

  • ☐ % users affected

  • ☐ Data loss / corruption?

  • ☐ Latency / error-rate impact

  • ☐ Financial or contractual impact

4) Timeline (Minute-Accurate)

  • ☐ First anomaly observed

  • ☐ Alert fired

  • ☐ Acknowledgement

  • ☐ Mitigation start

  • ☐ Full recovery

  • ☐ Follow-up actions start

Timeline must be factual, not interpreted.

5) What Failed (Technical)

  • ☐ Failing component identified

  • ☐ Triggering event identified

  • ☐ Primary failure mode

  • ☐ Secondary / cascading failures

  • ☐ Hidden dependencies exposed

  • ☐ Assumptions invalidated

6) Why It Failed (Root Causes)

  • ☐ Code defect

  • ☐ Configuration error

  • ☐ Capacity exhaustion

  • ☐ Dependency failure

  • ☐ Data inconsistency

  • ☐ Human error

  • ☐ Process gap

Multiple root causes allowed. Avoid “the” root cause fallacy.

7) Blast Radius Analysis

  • ☐ Why impact spread as far as it did

  • ☐ Isolation boundaries that failed

  • ☐ Controls that limited damage

  • ☐ Controls that were missing

8) Mitigation & Recovery

  • ☐ Immediate mitigation steps

  • ☐ Manual vs automated recovery

  • ☐ Rollback used?

  • ☐ Data repair required?

  • ☐ Time to stabilize

9) What Worked

  • ☐ Detection mechanisms

  • ☐ Runbooks

  • ☐ Tooling

  • ☐ Team coordination

  • ☐ Decision-making clarity

10) What Didn’t Work

  • ☐ Alerts ignored or unclear

  • ☐ Runbooks outdated or missing

  • ☐ Poor observability

  • ☐ Slow rollback

  • ☐ Communication gaps

11) Customer Communication

  • ☐ Status page updated?

  • ☐ Timing appropriate?

  • ☐ Message accuracy

  • ☐ Follow-up explanation delivered?

12) Corrective Actions (Hard Requirement)

Each action must be:

  • ☐ Specific

  • ☐ Owned

  • ☐ Prioritized

  • ☐ Trackable

  • ☐ Time-bound

Categories

  • ☐ Code changes

  • ☐ Architecture changes

  • ☐ Config / limits changes

  • ☐ Alerts / SLO updates

  • ☐ Runbook updates

  • ☐ Process improvements

13) Prevention Check

  • ☐ Would this incident happen again?

  • ☐ If yes, under what conditions?

  • ☐ Can it be auto-detected earlier?

  • ☐ Can it be auto-mitigated?

  • ☐ Can blast radius be reduced?

14) Learning & Signals

  • ☐ New failure mode discovered

  • ☐ Unknown dependency revealed

  • ☐ Capacity assumption invalidated

  • ☐ Organizational bottleneck exposed

15) Approval & Closure

  • ☐ Action items reviewed

  • ☐ Owners acknowledged

  • ☐ Follow-up review scheduled

  • ☐ Incident officially closed

Red Flags (If Any = Failed Postmortem)

  • ❌ No concrete action items

  • ❌ Single “root cause” narrative

  • ❌ Blame on individuals

  • ❌ No architecture or alerting changes

  • ❌ No follow-up tracking
