Unit Test

  • Goal: Test one isolated piece of logic (usually a function).

Pros:

  • Fast and reliable

  • Easy to test edge cases

  • No DB setup needed

Cons:

  • Doesnt guarantee real behavior with actual DB

Integration Test

  • Goal: Test multiple parts working together, like:

    • Service + DB

    • API route + service + DB

  • How: Run your real Prisma query, let it hit the database, and check results.

Pros:

  • Verifies real-world behavior

  • Detects config/connection/query errors

Cons:

  • Slower than unit tests

  • Needs DB setup and possibly cleanup

Summary

TypeScopeUses Real DB?When to Use
Unit TestSingle function (use mocks)Test logic independently and fast
Integration TestMultiple componentsValidate system works as a whole (e.g., DB logic)

Yes in backend apps like yours (Express + TypeScript + Prisma + Vitest), there are common setups to handle each test type efficiently. Here's a standard testing setup used by many backend developers:

Common Testing Setups by Type

1. Unit Testing Setup (Fast, Isolated)

  • Test runner: Vitest

  • Mocking: Use vi.mock() to mock Prisma, service layers, etc.

  • Folder: tests/unit/ or colocated as __tests__ next to files

  • Example File: service.test.ts

  • Tools:

    • Vitest for test runner

    • ts-mockito, sinon, or just vi.fn() for mocks

2. Integration Testing Setup (Real DB, No Mocks)

  • Test runner: Vitest

  • Database: Use a test database (e.g., SQLite or a separate PostgreSQL DB)

  • DB setup/teardown:

    • Run migrations (via prisma migrate or db push)

    • Seed known data if needed

    • Clean up tables before/after tests

  • Folder: tests/integration/

  • Tools:

    • Vitest

    • dotenv for test DB env vars

    • prisma for managing schema/data

3. End-to-End (E2E) Testing Setup (Real HTTP Calls)

  • Test runner: Vitest or Supertest (for HTTP assertions)

  • App: Launch the full Express app (in-memory or test server)

  • Folder: tests/e2e/

  • Setup:

    • Start the app programmatically before tests

    • Use supertest to send HTTP requests

    • Use test DB as above

  • Tools:

    • supertest (to hit HTTP endpoints)

    • prisma

    • Vitest
